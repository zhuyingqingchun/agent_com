# 2026中兴捧月大赛（初赛）- 智能代理赛题说明
## 一、赛题背景
自动点外卖、AI游戏代打……随着大模型技术的迅速发展，基于GUI（图形用户界面）的智能代理（Agent）越来越受到关注，能力范围持续扩展。

现有GUI Agent大多面向PC环境设计，而移动端受**算力、存储、功耗、实时性**等多重因素限制，对GUI Agent的实现方案与模型能力提出了更高挑战。

为推动移动端GUI Agent领域的技术进步与落地应用，本次大赛汇聚技术力量，探索高效解决方案，助力移动端GUI Agent发展。

## 二、赛题简介
### 核心任务
设计基于**多模态大模型**的GUI Agent，完成用户指定的各类任务。

### 开发要求
1. 参考大赛提供的基础Agent代码；
2. 严格遵循接口协议规范；
3. 自主设计、优化、重构更优秀的Agent框架，最大化发挥模型能力。

## 三、输入信息
Agent接收两类核心输入：
1. **用户指令**：自然语言形式的任务需求
   示例：我明晚要去北京出差，帮我定一个北京南站附近的酒店
2. **手机截图**：安卓手机界面图像
   示例：努比亚Z80 Ultra手机的桌面截图

## 四、输出要求
Agent需合理调用模型，结合用户指令与当前截图，输出**下一步操作动作+对应参数**。

模型原始输出格式、语言无限制，参赛选手需在代码中提取标准化结果：
- `action`：当前步骤需执行的操作类型
- `parameters`：操作类型对应的具体参数

### 合法动作与参数规范
| 动作类型（action） | 含义 | 对应参数（parameters） | 模型输出示例 |
|--------------------|------|------------------------|--------------|
| CLICK              | 点击 | point（坐标点） | CLICK:[[100, 200]] |
| TYPE               | 输入 | content（输入内容） | TYPE:['内容'] |
| SCROLL             | 滚动 | start_point、end_point（起始/结束坐标） | SCROLL:[[100, 200], [200, 300]] |
| OPEN               | 打开应用 | app_name（应用名称） | OPEN:['中兴管家'] |
| COMPLETE           | 任务完成 | content（无参数） | COMPLETE:[] |

### 坐标特殊要求
输出坐标为**相对坐标**：将图片宽高归一化至1000后的对应坐标，评测脚本会自动转换为实际图像坐标。

## 五、开发限制
- 开发语言：**Python**
- 环境版本：**Python 3.10.12**（必须严格使用）

## 六、赛题数据
1. **最终测试数据**：不同场景安卓手机截图（覆盖不同手机型号、不同App、不同界面）；
2. **调试支持**：主办方提供基础Agent代码、评测代码、部分测试用例，供选手调试；
3. **参考附件**：《捧月初赛-智能代理-选手参考附件_V1.2.zip》。

## 七、评分规则
### 评分流程
主办方运行评测代码，调用参赛Agent执行任务：
- 逐步骤检查操作及参数正确性；
- 成功执行至**COMPLETE完成状态**，本条任务计1分；否则计0分。

### 最终晋级计分公式
总得分 = 打榜得分×60% + Agent代码×20% + 设计方案及创新×20%

## 八、提交要求
### 提交格式
提交**压缩包（submission.zip）**，解压后所有文件总大小≤20MB。

### 压缩包目录结构
```
submission.zip
├── doc/
│   └── 算法设计说明文档
└── src/
    ├── agent_base.py    # 主办方提供，Agent基类，禁止修改
    ├── agent.py         # 选手实现，Agent派生类，必选
    ├── utils/           # 辅助代码，可选，可自由修改
    └── requirements.txt # 依赖Python库清单，必选
```

## 九、注意事项
1. 为保证代码行为一致性，**必须使用Python 3.10.12环境**；
2. 主办方提供的基础Agent代码仅为参考，在满足赛题限制前提下，鼓励扩展、改进、重构；
3. 严格遵循相对坐标规范（归一化至1000）；
4. 提交前，务必在本地运行主办方评测脚本，确认接口符合规范后再提交。
